# `0x10010000` BBC 文档树规范

本页说明 `~/gh/BBC` 最新规范在本仓库定向目录中的落地方式。

BBC 不是文件名前缀的字符串编号，而是 Markdown 目录、独立文件和标题共同组成的树形结构规范。

每个受 BBC 约束的节点都必须能还原到固定 8 位布局 `0xD1D2D3FH2H3H4H5`。

## `0x10011000` 本页结论

### `0x10011100` 需要修正的旧模型

#### `0x10011110` 字符串追加误区

##### `0x10011111` 验证信号

旧模型把 `0x100000-orientation` 和 `0x1100000-reading-model` 当作目录名。

这相当于把目录载体的 heading code 写进了文件系统名称。

最新版 BBC 规定目录名和独立文件名只能写 1 到 4 位 name code，也就是 `D1D2D3F`。

因此 `0x100000-orientation` 这种 6 位目录名不是合法 name code。

`0x10-orientation` 也不能表示第一层目录。

如果按 name code 解析，`0x10` 会先还原为 `0010`，再变成完整布局 `0x00100000`。

这个布局的目录位是 `001`，跳过了 `D1` 和 `D2`，不符合目录位必须左对齐连续的规则。

### `0x10011200` 正确模型

#### `0x10011210` 槽位恢复

##### `0x10011211` 最小规则

正确做法是先根据书写位置恢复规范布局，再在对应槽位解释数字。

目录和独立文件只在文件系统中写 `D1D2D3F`。

Markdown 标题写恢复后的 heading code。

H1 使用目录载体或独立文件本身的完整布局。

H2 到 H5 使用完整布局里的 `H2 H3 H4 H5` 低位槽。

标题序号不是追加到文件名编码后面。

本仓库把定向、BBC、源码快照、阅读路径、术语和维护收敛在 `0x1000-orientation`。

不再额外创建只承载这一组专题的 `0x1100-reading-model` 子目录。

## `0x10012000` BBC 完整布局

### `0x10012100` 八位槽位

#### `0x10012110` 结构职责

##### `0x10012111` 固定宽度

BBC 的完整布局是 8 位。

```text
0x D1 D2 D3  F-H1 H2 H3 H4 H5
   └───────────╂────────────┘
    Directory  ┃    Heading
           File and H1
```

`D1 D2 D3` 表示从文档根向下的目录路径。

目录位从左到右填充。

一层目录是 `a00`，两层目录是 `ab0`，三层目录是 `abc`。

`F-H1=0` 表示当前目录的 README 载体。

`F-H1=1` 到 `F` 表示当前目录下的独立文件。

同一个 `F-H1` 也是该载体或独立文件的 H1 位。

`H2 H3 H4 H5` 表示 H1 下方的标题路径。

BBC 不编码 H6。

### `0x10012200` 连续与稠密

#### `0x10012210` 子节点序号

##### `0x10012211` 禁止预留

同一父节点下的同级子节点从 `1` 到 `F` 稠密递增。

`0` 只表示该结构槽为空，不表示第零个节点。

目录位不能跳层。

标题位也不能跳层。

如果某个父级需要超过 15 个同级子节点，应拆出新的父级目录、独立文件或标题分组。

## `0x10013000` 书写位置

### `0x10013100` Name Code 与 Heading Code

#### `0x10013110` 恢复规则

##### `0x10013111` 位置优先

同一个 `0x` 字符串必须先看书写位置，再解释含义。

| 位置 | 写法 | 还原方式 |
| --- | --- | --- |
| `README.md` | 固定文件名 | 使用所在目录的 `F-H1=0` 载体完整布局 |
| 目录名 | 1 到 4 位 name code | 左侧补齐到 `D1D2D3F`，再追加 `0000` |
| 独立文件名 | 1 到 4 位 name code | 左侧补齐到 `D1D2D3F`，再追加 `0000` |
| Markdown 标题 | 1 到 8 位 heading code | 左侧补齐到 `D1D2D3FH2H3H4H5` |

目录名和独立文件名不能写 `H2H3H4H5=0000` 的完整标题码。

README 不消耗独立文件序号。

独立文件从 `F-H1=1` 开始编号。

### `0x10013200` 前导零省略

#### `0x10013210` 受控范围

##### `0x10013211` 一致宽度

前导零只能在一个受控范围内统一省略。

目录名和它的 README 标题属于同一个受控范围。

独立文件名和它正文里的标题属于同一个受控范围。

本章节的目录和文件 name code 都以非零 `D1` 开头，因此没有可省略的前导零。

## `0x10015000` 文件内标题算法

### `0x10015100` 独立文件示例

#### `0x10015110` 本页计算

##### `0x10015111` 填槽过程

本页文件名 `0x1001-bbc-encoding.md` 的 name code 是 `0x1001`。

它恢复为完整布局 `0x10010000`。

这个布局中的 `D1=1`、`D2=0`、`D3=0` 表示定向目录。

`F-H1=1` 表示该目录下第一个独立文件，并同时表示本页 H1。

第一个 H2 把 `H2` 槽从 `0` 改为 `1`，得到 `0x10011000`。

第二个 H2 把同一个 `H2` 槽改为 `2`，得到 `0x10012000`。

第一个 H2 下的第一个 H3 把 `H3` 槽改为 `1`，得到 `0x10011100`。

继续向下时，第一个 H4 是 `0x10011110`，第一个 H5 是 `0x10011111`。

### `0x10015200` 追加反例

#### `0x10015210` 错误布局

##### `0x10015211` 解析后果

把本页第一个 H2 写成 `0x100100001` 是错误的，因为 heading code 最多 8 位。

把本页第一个 H2 写成 `0x10010001` 也是错误的。

它会被解析为 `D1=1 D2=1 D3=0 F-H1=1 H2=0 H3=0 H4=0 H5=1`。

这等于在没有 H2、H3、H4 的情况下直接创建 H5。

正确 H2 是 `0x10011000`。

### `0x10015300` README 示例

#### `0x10015310` 目录载体

##### `0x10015311` 同一算法

`README.md` 的目录载体完整布局是 `0x10000000`。

它的第一个 H2 填入 `H2=1`，得到 `0x10001000`。

它的第一个 H3 填入 `H3=1`，得到 `0x10001100`。

README 使用 `F-H1=0`，所以不占用独立文件序号。

同目录第一个独立文件仍然是 `0x1001-bbc-encoding.md`。

## `0x10016000` 目录、文件与标题分工

### `0x10016100` 目录 README

#### `0x10016110` 文本承载

##### `0x10016111` 目录职责

每个文档目录都应有自己的 `README.md` 作为目录载体。

目录 README 负责说明该目录的定位、边界、子页面索引和推荐阅读入口。

目录 README 不负责承载具体专题的正文。

不要让目录下第一个独立文件替目录 README 承担目录说明职责。

### `0x10016200` 独立专题文件

#### `0x10016210` 问题边界

##### `0x10016211` 文件职责

独立专题文件负责一个具体问题、一条阅读链路或一个可验证的维护动作。

独立专题文件使用非零 `F-H1`。

它的 H1 与文件自身完整布局相同。

如果文本只是解释目录是什么、目录下有什么、应该先读哪个页面，它应进入目录 README。

### `0x10016300` 标题层级

#### `0x10016310` 表层结构

##### `0x10016311` 连续与稠密

标题层级表达文件内部的表层结构。

H2 到 H5 必须沿 `H2 H3 H4 H5` 连续填槽。

同一父标题下的同级标题从 `1` 到 `F` 稠密递增。

不能为了未来插入预留空号。

如果同一父级需要超过 15 个同级标题，应拆出新的父级标题或新的独立专题文件。

## `0x10017000` 维护与校验

### `0x10017100` 新增节点

#### `0x10017110` 决策顺序

##### `0x10017111` 放置规则

新增内容时，先判断它是在说明目录本身，还是在说明一个独立专题。

说明目录本身时，写入该目录的 `README.md`。

说明独立专题时，创建或更新 `0x<name-code>-<kebab-topic>.md`。

新增标题时，先找最近的父标题，再使用该父标题下一个可用十六进制序号。

所有标题编码都必须放在 Markdown code span 中。
